# AGENTS.md

**Per-attempt history:** each `attempt_N/README.md` snapshots the AGENTS.md state that attempt ran under — audit only, not read by the agent.

1. **Problem (writing this file):** keep AGENTS.md compact and outcome-named.
   - **Value:** compact, outcome-named files force the agent to re-derive the *how* each iteration from its tools and the corpus; without this discipline the file rots as the domain shifts and the agent over-fits to one iteration's specifics.
   - **Contract and constraints:** A problem is a self-amplifying attractor — one autonomous concern with a single extremum (its Reward) and a single trigger (its Attention mechanism) that pulls the agent's attention and amplifies until satisfied. Each is a concern the operator has offloaded to the agent (the agent watches, triggers, and drives it), making this file a delegation protocol, not a checklist. Only one problem is in foreground at a time (Ukhtomsky); before every action the agent verifies exactly one is dominant — concurrent problems signal confusion to resolve first. Control passes when a stronger trigger fires or the current extremum is approached; an interrupted problem is resumed, not restarted, so each persists enough state (trajectory files, addenda, in-progress search position) for resumption to be genuine. Problems are written for an intelligent agent searching a fuzzy environment with the available tooling in Ralph loops: supply no implementation detail the agent can fill itself, and — since humans reread this file often — keep each as short as possible. Every problem has five sections: **Value** (why it exists — would the project notice its removal?); **Contract and constraints** (the operator-only boundary the other sections optimize within — the agent does not edit it); **Solution search approach and hints** (what observed failures taught the agent to try first, plus this agent's own loop or trigger for returning to the problem); **Reward** (one extremum the operator reads off without judgement — a count, fraction, or comparable artefact); **Attention mechanism** (how a change this problem makes or notices in the shared environment becomes salient to whichever problem must react — the channel it publishes a signal to or watches, e.g. a log line, a metric, a `feedback.jsonl` entry, or an observability stream — as distinct from the agent's own re-entry loop). The one-liner names what the problem produces, not how. Backtick-pinned concretes (paths, command shapes, file and element names) are recognition scaffolding and survive trims; aging enumerations (versions, models, repo names) get stripped. The file is clustered — meta (P1); recipe and setup (P2–P4); substrate (P5–P9); external dependency (P10, observability consumed from the standalone `frog's-eye` repo, not built here); public distribution (P11); demand-driven application (P12 discovery/routing → P13 single-hop / P14 multi-step); cross-cutting principles (P15 no-pre-defense, P16 root-cause bug-fixing — peers of the meta-problem P1, numbered last to leave P2–P14 refs intact) — and within a cluster ordered by what serves what (the production rung-2 is a panel of off-the-shelf agents, so portability and cross-agent robustness are properties of P3, not separate problems). The open attempt is always the directory `current_attempt`; every tool references `current_attempt/...` (never a hardcoded number). An attempt gets its number only at archival: closing it renames `current_attempt` → `attempt_N` and leaves it frozen in place — so opening the next attempt needs no scripts, just that rename (no symlink to move). Attempt-agnostic shared helpers live in a frozen attempt (e.g. `attempt_7/tools`) only while never modified; the moment the open attempt must edit one — e.g. the build/test entry script `run_one_stage_v2.sh` or the `docker_phase`/`shallow_fetch` harness it reaches through — it is copied under `current_attempt/` and referenced sibling-relative (resolve siblings via `__file__`, never a hardcoded `attempt_N` path), so the tuned harness ships and is versioned with the attempt while frozen attempts stay byte-frozen.
   - **Solution search approach and hints:** read → why → intent — for each clause ask "why is this here?". Strip when the answer is mechanism the agent already fills, a bare plea for the rule, or an aged enumeration; keep when it scopes when the rule applies, names a recognition concrete, or signals when to revisit. A topic that always fires inside another problem's loop is mechanism of that loop — it belongs in that loop's Contract, not as its own problem. Re-run every editing pass; additionally, whenever the operator touches a problem's Contract and constraints, the next pass re-audits that problem's agent-mutable sections.
   - **Reward:** cuts that lose words without losing the rule or its scope; after the pass, every problem's Reward names something the operator can read off without judgement.
   - **Attention mechanism:** the file is the channel — an operator edit to a problem's Contract, or a problem's agent-mutable sections visibly bloating, is the signal that pulls this meta-problem to re-audit that problem.

## Recipe cluster

2. **Problem (Java LTS bump artifact for any coding agent):** ship the **`bump-java-version`** skill as **TWO steps** — (1) a **`detect-java-version` router** and (2) one skill **per LTS hop** (`bump-java-8-to-11`, `bump-java-11-to-17`, `bump-java-17-to-21`, `bump-java-21-to-25`). **Step 1 (router)** reads the project's *real* current Java level — its declared **bytecode target** across all modules (`options.release`/`<maven.compiler.release>` > `source/targetCompatibility`/`jvmTarget` > the build toolchain), NOT the toolchain the dig classifies by — and **either dispatches to the matching one-LTS hop or returns `NOT_A_BUMP`** for a project that should not be bumped (a deliberate-low-target library/plugin/instrumentation that pins bytecode below its toolchain, a multi-target build, or a `java-gradle-plugin`; bumping its bytecode would regress its consumers). **Step 2 (hop)**, given `(build files)` for its fixed `from→to`, performs the migration — preferring a declarative conversion program (unparametrized OpenRewrite recipes + predefined fixed intents) but allowed manual/parametric changes at a per-change penalty (see Reward) — which the harness applies and scores. (v2 program-synthesis pipeline; see `current_attempt/current_iteration/V2_PIPELINE.md`.) On each next-LTS bump (8→11, 11→17, 17→21, 21→25) it drives the highest corpus PASS rate, where a real bump means the project **targets** the new LTS (gains new syntax/APIs and CVE-free deps), not merely compiles under it. The router NEVER migrates (it only classifies + routes); the hop's purity is graded by the per-change reward, and GEPA optimizes each step's prompt independently.
   - **Value:** an empirical migration skill that beats the one-shot OpenRewrite baseline is the primary research goal; portability across agents is the audience constraint, so the deliverable is self-contained and harness-agnostic.
   - **Contract and constraints:** the deliverable is the `bump-java-version` skill — a single self-contained `SKILL.md`: a standard-tools-only hand manual (JDKs, Maven or Gradle, and OpenRewrite recipes pulled from Maven Central; no project-specific scripts) that migrates a Maven or Gradle project one Java LTS step so it **targets** the new LTS — under the new JDK it builds and conserves its previously-passing tests, its **effective compiler target is raised to `jv_to`**, and its resolved dependencies are **free of known CWEs** (so it gains new syntax/APIs and patched deps, not merely new bytecode); driven by a coding agent through the harness's sealed-env `bjv from/to` build/test/run tools over an **offline** Maven mirror + **frozen** CWE snapshot (no internet), all stage-agnostic (stage facts arrive via the consumer's harness as a prepended Stage header). Reads P4 for the stage list. Per-stage validation is delegated to P3, which returns `{terminal_rung, terminal_outcome, artifact_updates}`; fold `artifact_updates` into the corpus `bump-java-version` skill iff that does not regress corpus reward, else leave it unchanged and record the stage unintegrated. Concurrency across stages from P9. **Recipe-path precondition (fail-fast):** before migrating, the skill must verify the OpenRewrite recipe path actually resolves — the named recipe artifacts and their Maven-Central/proxy reachability — and if it cannot (proxy unreachable, recipe unresolvable), it **bails immediately with `RECIPE_PATH_UNREACHABLE`** rather than fall back to manual edits. A blocked deterministic path must surface as an explicit failure up front, never slide into improvised (hand-rolled) edits mid-run — the exact failure mode the OpenRewrite-purity reward and `I_MADE_MANUAL_EDIT` exist to forbid: an agent that, cut off from its recipes, fabricates a plausible-looking migration from its own head is producing unauditable output, not a bump.
   - **Solution search approach and hints:** when a failure family's stages only pass at the strongest rung (Opus) and won't harden down to the agent panel, that family lacks a corpus-level remedy — commission a recipe via P3's search or add a failure-output row. An `artifact_updates` that passes its own stage but regresses others on rollout is over-fit — generalise its trigger or scope the fix narrower. Loop continuously round-robin until reward stops improving, restarting from cached trajectories whenever the corpus artifact's sha materially changes.
   - **Reward:** corpus PASS rate (fraction of stages P3 terminates at the agent panel with `terminal_outcome=PASS`) vs the one-shot `UpgradeToJava<jv_to>` baseline. **OpenRewrite-purity (binary, per bump):** a bump scores **1 if every change was applied either by an *unparametrized* OpenRewrite recipe** or by a **predefined allowed intent** (listed below) — invoked by FQN at stock defaults (no per-project parameters, no custom recipe YAML), resolved from Maven Central, run via `rewrite:run` / `rewriteRun`; a recipe that needs project-specific parameters is a weaker, non-portable form. **Allowance is by intent, not by operation.** **Predefined allowed intents** (deterministic, auditable toolchain operations — *not* improvisation; operator-curated, extend by asking) — each counts toward reward 1 exactly like a recipe: (1) **update the build wrapper** — Gradle `gradlew`/`gradlew.bat` (or Maven `mvnw`) — to the version floor the target JDK requires; (2) **set the Java toolchain / compiler `release`** to `jv_to`. Any change that is **neither an unparametrized recipe nor a predefined allowed intent** — above all improvised source/dependency edits — is **never made**: the per-bump agent does not hand-edit and does not ask mid-run; it **bails with terminal code `I_MADE_MANUAL_EDIT`** (reported failed regardless of tests). The allow-list grows only **between runs** and only by the **operator**: the optimizing loop may *propose* a new predefined intent when bails recur, but **only the operator approves** adding one — whereas **fixed (unparametrized) recipes may be added to the skill freely** (inherently auditable). So the per-bump agent is fully deterministic — recipe or predefined intent → proceed, anything else → bail — and the skill improves by adding recipes (free) and operator-approved predefined intents. This drives every step, including the walls now handled by deterministic build-file edits, to be expressed as a **stock, reproducible, tool-applied recipe** rather than agent improvisation or a bespoke parametrization. **Open operator decision (CWE leg of the combined gate, P3):** clearing a known CWE (§5 of the skill) requires a dependency-version upgrade, which is inherently *parametrized* — so it currently scores the weaker tier, not purity-1. Whether **"upgrade a dependency to its nearest CWE-free version present in the mirror"** joins the predefined allowed intents (counting toward reward 1, like wrapper/target) is a pending operator call; until decided, CWE-fix upgrades are parametrized recipes (weaker tier) and `CWE_UNFIXABLE_OFFLINE` is the bail when the mirror has no fixed version. **Two-step reward (2026-06-20, supersedes the recipe-or-bail / `I_MADE_MANUAL_EDIT` framing above):** the skill is scored in two independent tiers matching its two steps. **(A) Router reward — `detect-java-version`, binary per repo:** **1** if the router produced a routing decision — it **dispatched a bump hop OR returned `NOT_A_BUMP`** — *without itself preparing or bumping anything*; **0** if it attempted any migration work of its own (writing a `rewrite.yml`/recipe, editing any build or source file, running a build to change the project). The router's only job is to classify the real bytecode level and route; doing the migration is out of role and zeroes it — regardless of whether the eventual bump would have passed. A correct `NOT_A_BUMP` is **full credit** (refusing an impossible/harmful bump is the right answer, not a failure); routing to the right hop is full credit; only stepping outside the classify-and-route role fails. **(B) Hop reward — the dispatched per-hop skill:** the **combined gate × 0.9^(model-chosen parametric recipes + manual edits)** — PASS the combined gate (builds+conserves under `jv_to`, effective target == `jv_to`, no gating CWEs) for the 1.0 ceiling, then ×0.9 per *injected* change: a parametric/project-specific recipe **or** a manual build/source edit. FREE (no penalty): unparametrized meta-recipes + the hop-fixed intents (set target to `jv_to`; pinned wrapper bump). Manual edits are now **allowed** (the `I_MADE_MANUAL_EDIT` bail is retired) but penalized, so the skill is pushed toward stock recipes without forbidding the edits real walls need; **test/assertion tampering still scores 0** (cheating). There is **no allow-list catalog** — any OpenRewrite recipe is permitted; a model-chosen parametric recipe simply costs 0.9 like an edit (so laundering an edit through `ChangePluginConfiguration` gains nothing). GEPA optimizes the two prompts independently against their two rewards.
   - **Attention mechanism:** `feedback.jsonl` is the channel — a failure family clustering there, or its stages only passing at the strong end of the rung ladder (stuck at Opus), signals that the family needs a corpus-level remedy.
3. **Problem (per-repo skill + recipe escalation):** for each (repo, commit) stage P2 hands down, harden the current best (skill, recipes) artifact until it drives the bump to PASS or labelled BAIL across a Qwen-driven panel of off-the-shelf agents — by making the stage pass on a ladder of executors, all driven by Claude — routinely two rungs: first Claude+Opus 4.8, then Claude managing the production panel — the same Qwen 27B run headless through three unrelated third-party agents (`opencode`, `kilocode`, `openhands`) in one unified harness (the JDK-less `bjv-controller` + sealed per-JDK `bjv-jvm<N>` images, the agent calling `bjv from/to`, never a JDK directly) where the agent is the only variable — with Claude patching the artifact at the stronger rung so it survives the weaker one. The panel replaces both the old single-agent (OpenHands) target and the separate portability + cross-agent witnesses: three stranger agents on the identical skill make portability inherent, and agent disagreement on a stage is the sharpest skill-clarity signal short of Opus.
   - **Value:** P2 sees corpus PASS/FAIL rates but cannot distinguish "stage genuinely unfixable", "production LLM too weak", "production harness wrong", and "artifact has a gap". This problem separates those for one stage: the routine rungs name most of the failure mode — fails even at Opus → genuinely unfixable or needs a new recipe; reaches Opus but fails the panel → the skill is unclear to a weak LLM or leans on one agent's tool-use conventions. The panel itself resolves the last ambiguity that used to need a separate bare-Qwen probe: fails under all three agents → the skill needs hardening; fails under one but passes the others → that agent's harness limit (surface to P8), and the cross-agent disagreement names the exact instruction to tighten. Because the three agents are off-the-shelf and run the skill with no assumption of our harness, the panel also witnesses portability (P2's no-assumption rule: it must pass under strangers' agents, not only inside one) — so a separate portability/cross-agent witness problem is no longer needed. It is also the path by which a single stage's skill/recipe insight propagates into the corpus artifact.
   - **Contract and constraints:** The (skill, recipes) artifact has a pinned shape so every rung reads the same structure and a failure maps to a known section. The skill is a single self-contained `SKILL.md` — a standard-tools-only hand manual (JDKs, Maven or Gradle, and OpenRewrite recipes from Maven Central; no project-specific scripts) — whose numbered sections are pinned in order: Tools; record the baseline under `jv_from`; make Lombok safe (a **proactive step**, see below); run the OpenRewrite migration (recipes by FQN); apply the deterministic JDK-removal fixes (via recipe/intent, not hand-edits); under the sealed `jv_to` env build + test + conserve + confirm target==`jv_to` + CWE-scan (`bjv to build/test/scan`); the Spring Boot upgrades a Java bump forces; a Troubleshooting table (failure → remedy); and When to bail (the live instance is at `.agents/skills/bump-java-version/SKILL.md`). **Proactive steps vs. reactive Troubleshooting (2026-06-22):** the skill distinguishes two fix kinds. A **reactive** Troubleshooting row fires *on an observed error* (read the error → apply the named remedy). A **proactive step** runs *before the first build under `jv_to`*, gated on a **structural** signal (what the project *declares/uses*), never on error text. A wall must be made proactive when its failure is **open-ended and self-obscuring** — its compile symptoms vary per project and never name the real cause, so reactive error-matching is unreliable. **Lombok is the canonical proactive step (live for 21→25; applies to every hop):** an out-of-date Lombok silently stops generating members under a newer JDK, surfacing as *open-ended, Lombok-unnamed* downstream errors (`cannot find symbol getX()`/`builder()`/`log`, `constructor … cannot be applied`) whose exact symbols differ per repo — and the *same* `cannot find symbol` is also emitted by genuine non-Lombok API changes, so the error text cannot identify Lombok. The reliable discriminator is structural: *does the build declare `org.projectlombok:lombok`?* If yes, **floor Lombok to the JDK-safe pinned version AND force annotation processing before building** (21→25 → lombok **1.18.46** — older releases predate JDK 25 — *and* force the processor to run, since JDK 23+ no longer auto-runs a classpath-only processor: Maven `maven.compiler.proc=full`, Gradle the annotationProcessor path), never waiting for a symptom; the matching reactive Troubleshooting row stays as a back-stop. Because the floor is a single **hop-pinned** version (not a project-chosen one), it is a candidate to join the free hop-fixed intents like the wrapper/target bumps — an **open operator decision**; until promoted it scores as a parametric dependency-floor (×0.9) under the hop reward. New proactive steps are added by this same test: open-ended/ambiguous symptoms + a reliable structural trigger. Two pins a reader needs here: the Troubleshooting table's fix vocabulary is {OpenRewrite recipe by FQN run via `bjv to/from run` (`mvn rewrite:run`/`gradle rewriteRun` inside the sealed env), a predefined allowed intent (wrapper bump / set target to `jv_to`), a CWE-fix dependency-upgrade recipe, docker cleanup, labelled bail, ignore-with-justification} — never a hand-edit, and the Reward (PASS) is the **combined gate**, all evaluated in the **sealed `jv_to` env** via `bjv to …`: (a) **builds and conserves** — `bjv to build` succeeds AND pre-pass test set ⊆ post-pass test set (`bjv to test`); (b) **target raised** — the effective compiler target (source/`release`/toolchain) is `jv_to` (a project left below `jv_to` that merely compiles is NOT a bump — the soft-pin false pass); (c) **CWE-clean** — `bjv scan` reports no known CWEs in resolved deps (offline frozen snapshot; the exact gating threshold is TBD pending measurement — reported before it gates). All three are required: a test-conserving compile that leaves target < `jv_to`, or ships a known-CWE dependency, is a FAIL. A baseline with **no passing tests** is not discarded but **split**: one that fails to compile under `jv_from` is unscorable (`NO_BASELINE_NOCOMPILE`); one that compiles under `jv_from` (`NO_BASELINE_NOTESTS`) is scored on **compile alone** — a migration that still compiles under `jv_to` is a **compile-only PASS**, a weaker success tier than a test-conserving PASS and reported separately, because a project that migrates and still builds is a real positive outcome even with nothing to conserve. Recipes are whatever the manual names by FQN, resolved from Maven Central (public OpenRewrite recipes); there is no bundled recipe catalog. Stage facts (repo URL, sha, jv_from, jv_to, workdir) are injected by each rung's harness as a prepended Stage header, never in the `SKILL.md`, so the skill stays portable. The search is one schema: per stage, the **search results** — the artifact version run at each rung (`opus`, then the agent panel `opencode`/`kilocode`/`openhands`) with its outcome — and the one **best search result**, the hardened artifact that passes the panel. Claude is the sole searcher at every rung: it runs the current artifact under that rung's executor and, on failure, patches the artifact (rewrite any `SKILL.md` section, add or reorder Troubleshooting rows, add a recipe FQN, tighten or reorder a step); the executor (Opus 4.8, or the Qwen 3.6 27B FP8 proposer — served from P7's `/qwen-3.6-27b-fp8/v1` route on GPU0, `max-model-len` 262144 (256K) — run through any panel agent) only runs what Claude ships; **v2 execution (current; supersedes the interactive panel/sealed-hop model):** the pipeline in `current_iteration/V2_PIPELINE.md` — an orchestrator (OpenHands restricted to 4 custom tools, NO bash/docker) calls `generate_program` (emits the recipe+intent plan), `check_program` (static anti-cheat: every op a catalog recipe or allow-listed intent, else reject), and `run_and_score` (applies the plan and scores the combined gate). Execution is a **single all-JDK image** `bjv-alljdk` (JDKs 8–25 + mvn + gradle + offline osv, `installations.paths` exposing every toolchain), with `JAVA_HOME` pinned per step — safe as one shared env because the bytecode `effective_target` gate, not the env, enforces the target. **Optimization (v2): GEPA** evolves `generate_program`'s prompt + the recipe catalog against the corpus, using the combined-gate result as the metric and the `run_and_score` logs + `check_program` violations as reflective feedback, with Pareto selection across repos/hops to resist overfitting. (The retired interactive loop instead had Claude patch the `SKILL.md` across an Opus→panel rung ladder.) A rung's FAIL is re-run-confirmed (flakes re-run at the same rung, not escalated); a run terminates when the combined gate PASSes or `generate_program` emits an honest bail / no-improvement. **No budgets or limits on the per-bump agent (RLVR).** Never cap its steps, wall-clock, tokens, or impose any timeout/budget on its work — and never let it face an offline resource it will hang on. In an RLVR setting a budget or limit teaches the policy to **give up and emit random noise** instead of persisting, so the agent runs **uncapped** and is expected to go **as deep as the task needs**; the only termination is verified-done or an honest labelled bail. A hang (e.g. a Gradle distribution download with no offline source) is an **infrastructure bug to fix at its root** — pre-stage the resource — never to bound with a timeout. Hence the hop harness has **no iteration cap** (`oh_run.py` `max_iteration_per_run` is effectively unlimited) and **no per-job timeouts**, and every build input (deps via the Nexus mirror, Gradle distributions pinned to versions staged in both classifiers) is pre-staged offline. **The same no-silent-hang principle covers the agent's LLM link to the shared vLLM:** it runs with **TCP keepalive** (`AIOHTTP_SO_KEEPALIVE=true`, `TCP_KEEPIDLE=60`/`TCP_KEEPINTVL=30` on litellm's aiohttp transport, exported by the hop driver before `oh_run.py`) so a slow-inference idle period — common when many agents share one busy vLLM — is **not silently reaped by the caddy/NAT proxy hop into a stalled socket** (which would otherwise burn the full request-timeout window before erroring and retrying). Keepalive is a connection-liveness probe, NOT a work-bounding limit — it detects a genuinely dead connection in ~60–90s instead of letting the agent wait on a corpse. Per repo×hop, `run_and_score` writes `out/<slug>/result.json` (verdict + `pre_pass`/`post_pass`/`lost` + `effective_target` + `dep_cwes`) alongside the emitted `program.json`, `check.log`, `apply.log`, build/test/scan logs, and `generate.log` — these logs are GEPA's reflective feedback for `generate_program`. Each rung's session is isolated (own workdir, conversation id, surefire path); concurrency bounded by P9. Output to P2: `{stage, terminal_rung, terminal_outcome, artifact_returned}`, where `terminal_rung` is the weakest rung reached at PASS or labelled BAIL (ideally the full panel) and `artifact_returned` is the hardened artifact; P2 decides folding.
   - **Solution search approach and hints:** fast-fail at a rung — if Claude+executor stalls (no progress for >N actions on the same step) treat it as FAIL at that rung, don't spin; never recurse on flakes (one-off network error, Docker glitch) — re-run the same rung. On a FAIL, Claude's first probe is to read the executor's actual trace (the agent transcript) to find where its reasoning diverged from the skill — typically a missed trigger paraphrase, a skipped step, or a tool-use convention the `SKILL.md` didn't anticipate. Claude's patch is the smallest possible (a one-sentence tightening, a row reword, a step reorder), not a wholesale rewrite; add a new recipe only when no prose change could fix the executor's behaviour. After patching at Opus, re-validate on the panel, since a fix can regress. A stage that keeps failing the panel after patching: if it fails under all three agents it's a genuinely unclear skill — harden it; if it fails under one agent but passes the others, that's that agent's harness limit (surface to P8, not a skill rewrite), and the cross-agent disagreement names the diverging instruction to tighten. Portability drift (the skill coming to depend on one agent's scaffolding) shows up automatically as a widening cross-agent gap, so no separate portability probe is needed. Re-invoke per stage each pass from P2, and again when the corpus artifact's sha changes and the stage previously did not pass the panel. Conservation (`pre-pass ⊆ post-pass`) is matched **rename-robustly**: agent test refactors (JUnit5 `@DisplayName`/`@Nested`, Kotlin backtick names, class renames) change the surefire `classname#name` key, so an exact set-difference over-reports lost tests — match the residual by normalized method name (component after `#`, `(params)`/`[index]` stripped) after consuming exact matches, as `agent_drive_one.sh` does, else legitimate renames misfire as `FAIL_test_conservation`.
   - **Reward:** fraction of input stages where the agent panel reaches PASS or labelled BAIL — both initially and after each patch; the metric is "panel success rate", not "any-rung success rate". Secondary: average patches per stage to reach panel success (lower is better).
   - **Attention mechanism:** `feedback.jsonl` is the channel — a re-run-confirmed FAIL trajectory written there is the signal that pulls Claude to patch the artifact at that rung; the panel still failing after patches (under all agents) signals P8 (a structural harness limit), not another skill rewrite.
4. **Problem (dataset = repo list + per-run randomized-baseline sampler):** maintain `dataset-repos.json` (repo names only); draw the eval set fresh each round by sampling, per repo, a random commit that declares Java 8/11/17/21 (via `pom.xml` or `build.gradle`/`.kts`) AND **runs at least ~5 passing tests under that JDK** (a real conserve-set — not merely compiles), and feed P2 by **round-robin across the four hops** so the eval stays evenly distributed by hop until the smaller queues exhaust.
   - **Value:** a fixed-sha corpus lets P2 (and its GEPA/EvoSkills optimizers) overfit to specific commits, and biases toward already-prep'd states that score high without real migration work. Sampling a *different* valid baseline per run makes the eval a moving target — only improvements that generalize across each repo's history survive. It also retires baseline curation: validity is checked when a baseline is drawn, not maintained as a clean-list.
   - **Contract and constraints:** the persistent dataset is `dataset-repos.json` (a list of `owner/repo`). Each round, `current_sweep/draw_iter.py --seed <round>` draws `dataset-shas.json` **afresh** (never a fixed upfront dataset) — **N = 100 datapoints/round** — by **randomly** sampling baselines from the quarter-structured baseline store (`corpus/baselines_peryear.json.jsonl`, the dig output), **≤1 sha per repo** so the eval stays repo-diverse and is a moving target across rounds. Entries `{repo, sha, jv_from, jv_to, year}` where `jv_from ∈ {8,11,17,21}` and `jv_to` is the next LTS (8→11, 11→17, 17→21, 21→25), **derived from `jv_from`**. Every baseline in the store was **verified to run ≥ ~5 passing tests under `jv_from`** at dig time (test-compile alone — and the static `@Test` count — admitted no-test / infra-only repos whose only "test" is a `@SpringBootTest contextLoads` that can't start offline, so it conserves nothing; the green-test gate excludes them). The sweep draws the per-hop queues **round-robin, balanced on completed datapoints** (each bite goes to the hop with the fewest scored so far), so the eval is even across all four hops early and tilts toward the larger queues (17→21, 21→25) only after the smaller (8→11, 11→17) run dry. The continuous 100-run iter loop is executed by a **single agent — OpenHands** — not the full 3-agent panel; the panel (P3, `opencode`+`kilocode`+`openhands`) is reserved for targeted rung-2 cross-agent validation (a 100×3-agent round is ~15h, too slow to run every iter). One repo yields several baselines — one per commit-**quarter** on the main branch (default), or one per Java version (`--multi`). Repos are sampled **most-stars-first** (the sampler's repo list is star-sorted before sharding), so a bounded run keeps the highest-signal baselines. P2 reads the per-run file. A different `--seed` ⇒ different shas; the same seed is reproducible. A repo that yields no compiling 8/11/17/21 baseline within the attempt bound contributes nothing that round — never a false `baseline-broken` into P2. Junk (already ≥ `jv_to`, no `pom.xml`/`build.gradle`, non-compiling) cannot enter the dataset because the sampler rejects it.
   - **Solution search approach and hints:** bounded random search, **per commit-quarter on the main/default branch** — bucket main commits by quarter; for each quarter randomly try up to `--max-attempts` (~2) commits, cheap-rejecting (no `pom.xml`/`build.gradle`) for free; the first that **runs ≥ ~5 passing tests** (`mvn test` / `gradle test` under its **derived** `jv_from`, counting tests that actually PASS — not the static `@Test` count, which overcounts badly: many "tests≥3" repos run 0 green at runtime — demo/reproducer projects, or a single infra-dependent `contextLoads`) is that quarter's baseline, and a quarter with none in budget yields nothing. So one repo gives one baseline per buildable quarter on main — its real migration arc — each tagged with `quarter`. **Runnable-passing tests ≥ ~5 (not test-compile) is the validity bar** so P2 never inherits a baseline with nothing to conserve. Favour the **high static-count band** (corpus `tests` 10–50) when sampling — that's where the green-test yield is high (~64%, median ~40 green tests); the low band (static 3–5) is almost all no-test junk (≈0% yield). Sampling across **full** history organically surfaces 8→11 and 11→17 hops, not just 17→21 — fetch unshallowed so the walk sees real ancestry. Keep `dataset-repos.json` broad (distinct owners, dependency-family spread); for a repo that rarely yields a baseline, raise `--max-attempts` rather than hand-curating. Use a fresh seed per optimization round and score edits on the **mean across seeds**, discarding seed-specific gains. The fresh, quarter-structured baseline store is the dig output (`{repo, sha, jv_from, quarter}`) over `corpus/discovered/all_candidates.txt` (the legacy yearless `attempt_db.json` was retired); draw the per-run **iter-db** (`dataset-shas.json`, the sweep's active corpus) from it with **≤1 sha per repo** so the eval stays repo-diverse, while the store keeps the per-quarter multiples for volume. Repo discovery itself is the scriptless **`discover-bump-dataset`** skill (gh code-search: Maven `pom.xml` and Gradle `build.gradle`/`.kts` declaring a target jv, deduped against the known corpus); its fresh candidates land in `current_attempt/corpus/discovered/{maven,gradle}_j<jv>_fresh.txt` and feed the sampler's `--repos-file`.
   - **Reward:** sampler hit-rate (fraction of `dataset-repos.json` that yields a valid compiling 8/11/17/21 baseline per run) and the LTS-hop diversity of the sampled set; and, for the optimizer, **low variance of P2's score across seeds** — a genuinely general artifact scores about the same on any sampled baseline set, an overfit one does not.
   - **Attention mechanism:** the per-run `dataset-shas.json` and the spread of P2's score across seeds are the channel — a repo that never yields a baseline (raise its attempt bound or drop it from `dataset-repos.json`), or a P2 gain that evaporates under a new seed (an overfit edit — reject it), is the signal that pulls P4 to adjust sampling.

## Substrate cluster

5. **Problem (proxy dependency resolution):** every build's external-artifact resolution goes through a local caching proxy with plural upstream mirrors.
   - **Value:** without a caching proxy, build failures get attributed to upstream availability blips rather than code state, polluting the parent loop's reward signal.
   - **Contract and constraints:** the proxy caches every artifact it serves and survives across iterations. Consumers (P2, P4) attribute build failures to code state, so an unresolved artifact triggers a proxy widening before that build counts toward reward.
   - **Solution search approach and hints:** include both live and archival mirrors; container builds reach the proxy by container-network DNS, not host IPs; when a build fails on "cannot resolve X", widen the upstream set before treating the failure as code-attributable. Re-enter whenever a parent loop's failures cluster on artifact resolution. A reverse proxy (Caddy) in front of an ANONYMOUS-read backend (Nexus) must STRIP the client Authorization header on the upstream hop (`header_up -Authorization`) — else the backend treats the edge token as one of ITS users, fails to find it, and 401s (realm in the WWW-Authenticate names the real rejector). Audit that the harness mirror actually points at the proxy: a settings.xml mirroring `central` straight to repo.maven.apache.org silently bypasses Nexus (found + fixed 2026-06; mirror must be `*` -> the maven-public group).
   - **Reward:** per-artifact cache-hit ratio (higher is better; resolution failures become distinguishable from compile failures in the parent loop's classification).
   - **Attention mechanism:** build logs are the channel — a `cannot resolve X` line is the signal that pulls P5 to widen the upstream set before that failure is counted as code-attributable.
6. **Problem (maintain local environment):** known stable paths, scratch-space discipline, and disk/cache stewardship for every other problem.
   - **Value:** iteration outcomes must vary only with what's being measured; unstable paths, shared scratch, and contention drift the noise floor up, and one problem exhausting a shared resource crashes another and corrupts the parent loop's results.
   - **Contract and constraints:** project workdir at `$HOME/java_8_11_17_to_java_21` (the one path pinned here — every other is an env var resolved from `.env` so the value lives next to credentials). Env vars consumers read: `MIRROR_CACHE_DIR` (per-repo git mirrors as `<owner>/<repo>.git`); `SCRATCH_ROOTS` (colon-separated writable mounts by preference; agent picks the first with headroom); `SCRATCH_SUBDIR_TEMPLATE` (relative template per scratch dir, e.g. `tmp/<repo_name>/<ff_specific_subdir>/`). **Every action is docker-bounded** — all execution (build toolchains, recipe runs, AND any ad-hoc scripting or analysis the agent performs) runs inside a container; the host is only a Docker host and a bind-mount source for data/cache dirs, never a place to run processes directly (no host `python3`/`mvn`/`gradle`/`sed`-on-data — run it in an image that already has the interpreter); SSH calls to the work host share one session, not one per command; free host resources held by unrelated work when this project needs them. **Docker container logs are rotated** — the daemon's default `json-file` driver has no rotation, so an unbounded container log (one reached 55 GB here) silently fills the disk and crashes dependent services (e.g. P10's compactor on its next write); `observe_rotate.py` truncates over-cap container json-logs via a root container, and the permanent fix is docker daemon `log-opts` (`max-size`/`max-file`) applied on a maintenance restart. A problem declares a scratch-dir name; this problem resolves it against `SCRATCH_ROOTS` + `SCRATCH_SUBDIR_TEMPLATE`. Each run reaps its own scratch workdir on completion (in a `finally`); because the build container writes root-owned files, the reap runs through a root container, not host `rm`. A scratch dir that outlives its run is a reap failure to investigate. The dig (P4, admit-time) and the conversion panel (P3, run-time) compile in **one shared build image** so a baseline that compiled when admitted also compiles when converted — a divergent image silently turns admit-green baselines into `NO_BASELINE_NOCOMPILE`; that image carries a **UTF-8 locale** (`LANG=C.UTF-8`), since the ASCII default leaves the JVM unable to write class files for non-ASCII identifiers (e.g. Kotlin backtick test names).
   - **Solution search approach and hints:** when a pattern repeats and is slow, pin a known location or cache; when a pattern risks an external rate-limit, cache the upstream once; never let docker-owned root-rw files leak — each run cleans its own scratch through a root container (not host `rm`) in a `finally`, so cleanup is never skipped or deferred. Re-enter whenever a new noisy or slow pattern emerges, or when a run's scratch survives the run — that means the per-run reap failed, so fix the reap. To run an ad-hoc script docker-bounded: pipe it on stdin into a stock image that has the interpreter — `docker run --rm -i -v <datadir>:<datadir> python:3-slim python3 - <args>` — mounting only the data dir **at its real path** so absolute paths resolve; a file a root container wrote is read/edited/reaped by a root container, never by a host tool. If the host ever looks like the path of least resistance for a quick script, that is the smell this rule exists to catch. A build container that outlives its harness timeout is the container flavor of a failed reap: a host-side timeout kills the `docker run` *client*, never the container — so a hung build (e.g. a surefire JVM waiting forever on a broker) survives for days holding shared-cache locks (`Timeout waiting to lock journal/artifact cache` storms in the observability stream, workers serialized, CPU idle). Bound the build *inside* the container (`timeout -k 15 $INNER_TIMEOUT <build>` in the wrapper's inner command, set just below the host kill window) so the container self-exits; long-lived build containers (the sealed `bjv-jvm<N>` images, spawned per `bjv` call) in `docker ps` are the canary.
   - **Reward:** iteration variance from non-measured factors approaches zero — zero ban incidents, predictable per-iteration wall-clock, locations re-used across iterations, zero disk-full crashes, bounded scratch-dir count per repo (a leak triggers investigation, not just cleanup).
   - **Attention mechanism:** host metrics and scratch-dir counts are the channel — a scratch dir that outlives its run, or a per-repo scratch-dir count creeping up, signals that the per-run reap failed and P6 must fix it.
7. **Problem (vLLM spin-up):** stand up tool-capable OpenAI-compatible chat-completion endpoints callable from inside Docker, with authentication enforced.
   - **Value:** P2's agent backend and P10's compactor cannot run without a tool-capable model endpoint reachable from inside docker containers; this problem owns that contract surface.
   - **Contract and constraints:** all endpoints in active use serve models from the same generation family — never mix generations, since prompts calibrated against one drift on another. When two consumers have meaningfully different throughput/depth tradeoffs they get separate endpoints on separate accelerators so they don't contend for slots. A single public gateway (`inference.mikhailov.tech`, a Caddy reverse proxy) fronts both endpoints — `/qwen-3.6-27b-fp8/v1` reaches the FP8 proposer (GPU0) and `/qwen-3.6-35b-a3b-awq/v1` the AWQ observability compactor (GPU1, container `obs-vllm`) — **served & owned by the external frog's-eye repo** (`~/frogs-eye`), not stood up here; the gateway just routes to it — the path names the served model; one hostname and credential, so consumers never hardcode internal container IPs (which churn on daemon restart), and unmatched paths return a 404. Each served model runs with both a `--tool-call-parser` and a `--reasoning-parser` matched to its family (e.g. `qwen3` for the Qwen3.x line), so tool calls and chain-of-thought are split into the response's `tool_calls`/`reasoning` fields and never leak into `content` — the chat template injects the opening `<think>` into the prompt, so it is the parser (not the template) that separates the thinking; a missing `--reasoning-parser` leaves a dangling `</think>` in `content` with `reasoning:null`. Per-consumer model identity and call-time options (thinking mode, etc.) live in the consumer's own Contract.
   - **Solution search approach and hints:** arrive at a working endpoint set through a ralph loop over container, model, and reverse-proxy config; converge by what consumers can actually call, not what the endpoint exposes in isolation. Re-enter on any consumer reporting degraded service.
   - **Reward:** consumers report uninterrupted service.
   - **Attention mechanism:** consumer call results are the channel — a consumer logging a degraded or failed completion is the signal that pulls P7 to re-check that consumer's endpoint.
8. **Problem (agent runtime spin-up):** stand up the configured agent runtime as a credentialed service reachable from the work host, with conversation lifecycle, event-stream emission, and a context-management primitive configured.
   - **Value:** P2's recipe search needs an agent runtime that accepts a prompt, executes against a working tree, emits structured events the harness consumes, and survives the deepest stages without losing trajectories to context overflow; this problem owns that surface.
   - **Contract and constraints:** the runtime exposes a callable interface (in-process SDK, HTTP API, or MCP socket) reachable from the harness, with tool invocation, conversation state across iterations, and event streaming. It is configured with at least one context-management primitive — an automatic condenser that folds older events into a summary at a threshold, or delegated subagents with their own context windows — so single-call context budget does not scale with repo depth and overflow is structural, not catastrophic. Every exposed surface shares P7's credential discipline: no unauthenticated bind, no credentials in trajectory state or in scratch dirs reaped by P6. Sub-context activity is captured in the event stream (P10) so the operator can audit when context was folded. The runtime has network reach to P5's caching proxy (same docker network or equivalent route) so every Maven invocation resolves through the cache, not direct upstream — without this, wall-clock balloons and upstream hosts ban us. Runtime identity is pinned per attempt so P2's PASS rates are comparable across passes. The runtime accepts N concurrent isolated sessions (own conversation id, workspace, event-stream slice), N dictated by P9; sessions share the LLM endpoint and proxy but never a workspace.
   - **Solution search approach and hints:** prefer maintained frameworks over reinventing tool-call iteration, context pruning, or conversation state; choose a runtime by what P2 can actually drive, not by what it advertises. Network reach goes through the same reverse-proxy + auth surface as the vLLM endpoints; in-process SDK use is permitted but not exempt from that credential discipline. If the agent stalls or repeats actions on the same failure across iterations, the missing input is usually structural source detail that needs sub-context — not a different prompt. A reasoning (thinking) model with native tool-calling has a distinct stall: **empty `tool_calls` + `finish_reason: length`** because thinking spent the whole output budget before emitting the call. Bound it with **`thinking_token_budget`** in the runtime's `extra_body` AND **`drop_params=False`** — litellm silently strips the unknown budget param off the wire otherwise (it keeps `chat_template_kwargs` but drops `thinking_token_budget`), a trap that only bites multi-turn/long-input, so a single-turn replay hides it. The endpoint half (splitting reasoning out of `content`) is P7's `--reasoning-parser`. Re-enter on any P2 stage reporting an agent-runtime failure unrelated to prompt or repo state, whenever context overflow drops a trajectory, or whenever a stage cluster shows the agent stalling without obvious cause.
   - **Reward:** P2 reports uninterrupted agent execution; fraction of stages where context overflow caused trajectory loss approaches zero; zero credentialless surfaces.
   - **Attention mechanism:** the agent event stream is the channel — a trajectory dropped to context overflow, a stall repeating actions on one failure, or an unauthenticated surface, signals P8 to re-examine the runtime.
9. **Problem (runner saturation):** keep every host compute resource (CPU, memory, GPU compute, GPU memory, disk I/O, network) in a healthy utilisation band while any parent loop is making progress; the binding constraint shifts across the loop's phases and is re-detected, not assumed.
   - **Value:** an idle host wastes budget; any over-saturated resource drops the loop's wall-clock-per-PASS reward to noise. The binding constraint is rarely CPU alone — under N-way agent parallelism it shifts from CPU at startup to GPU compute during agent rounds to disk/network during mvn cache warming to GPU KV-cache memory under sustained batching.
   - **Contract and constraints:** composes with the parent loop rather than standing alone — saturation decisions are conditional on which loop is active, its recent action history, and which resource is nearest its band edge; any concurrency dial the agent can reach is in scope. The primary output is a per-loop concurrency recommendation N (read by P2 to size its per-pass session fan-out into P8's runtime); N may shift mid-pass as the binding constraint shifts.
   - **Solution search approach and hints:** sample every relevant resource periodically (host CPU + load avg, host memory, per-GPU utilisation + memory + KV-cache from the vLLM logs, disk I/O on the work mount, network to P5's proxy and P7's endpoint, docker daemon health); learn the band per resource; re-detect the binding constraint when the loop changes phase. GPU compute-percent snapshots are bursty under batching — read the average over a window, not the instant. Run continuously, dampened against oscillation; pause when no parent loop is active.
   - **Reward:** sustained healthy band across every relevant resource without thrashing or stalling the parent loop; the binding constraint correctly identified at any moment.
   - **Attention mechanism:** the resource samples are the channel — any resource (CPU, GPU compute/KV-cache, disk I/O, network, from `host_metrics` + the vLLM logs) nearing its band edge signals P9 to re-detect the binding constraint and re-issue N.

12. **Problem (discover & route Java-bump demand):** continuously discover open GitHub issues asking for a Java-version bump (unsatisfied requests), triage them, and route each to P13 (single LTS step) or P14 (multi-step) — turning live demand into a prioritized, pre-vetted work queue. This problem finds and hands off; it does not fork, bump, or open PRs.
   - **Value:** corpus PASS rate (P2) proves the skill in vitro; a *merged* PR closing a request someone filed is ground-truth adoption. Owning the demand surface here — find, triage, dedup, prioritize, route — means P13/P14 each receive a clean already-vetted target and never re-derive the feed. A request the skill cannot satisfy is a corpus gap fed back to P2. And a merge is not the only win: every opened PR **leaves the skill's credit in the wild** — a footer linking the [`bump-java-version-skill`](https://github.com/vasiliy-mikhailov/bump-java-version-skill) repo — so each demand PR doubles as a real-world distribution/awareness channel for the skill (a P11 overlap). Leaving that credit is a first-class purpose of P12, not decoration.
   - **Contract and constraints:** the feed is open issues whose **title** asks to upgrade/bump Java/JDK (`state:open type:issue language:java`), deduped per repo — `tools/find_bump_issues.py` -> `p12/feed/bump_issues.json` (genuine bumps only; dependency-name / API-migration false positives dropped). The feed is re-run periodically (demand accrues over weeks, not hours — a daily re-run is wasted), and a **merged** PR removes its request automatically (the `resolves #N` closes the issue, dropping it from the open-issue search — observed with `mars-sim#1959`).    - **Solution search approach and hints:** prioritise high-star, recently-active requests; the clean single-step tail exhausts fast (the funnel collapses ~161 total → ~22 actionable), so multi-step (P14) and periodic re-discovery are where fresh high-value targets live. Re-enter on a feed refresh or a routed target's terminal outcome.

13. **Problem (single-hop demand PR):** for a one-LTS-step request routed by P12, run the `bump-java-version` skill exactly once, conserve tests under the repo's own CI command, and open ONE PR that fully resolves the request.
   - **Value:** the common case — most requests are one LTS step from the repo's current version, and a clean single hop is the cheapest, most-trustworthy migration to land. Kept distinct from P14 so the simple case carries no multi-hop machinery.
   - **Contract and constraints:** input is a P12-routed single-step request (`current_jv` → next LTS = `target_jv`). Fork to the operator's account; run the skill one hop; score with P3's conservation criterion — `mvn compile` under `jv_to` AND pre-pass tests subset of post-pass tests — **under the repo's own CI/build command** (`mvn verify`, read from `.github/workflows`), never a lighter `mvn test`, which can mask failures (a test-listener that silently fails to instantiate hides the real break, as `carml#628` showed — BlockHound's failure masked an `--add-opens java.base/java.lang` strong-encapsulation failure); where the repo's CI genuinely *is* `mvn test`, that is the correct gate (e.g. a release-only `gpg:sign` makes `verify` fail on something CI never runs, as `logback-elasticsearch-appender#48` showed). 14. **Problem (multi-step demand PR):** for a two-or-more-LTS-step request routed by P12, chain the skill hop-by-hop to the requested target, conserving tests at **every** hop, and open ONE PR that resolves the full jump.

Third-party-PR go-ahead, scorecard append (hop chain in the hop column), and no-churn rules are P13's. Reuses P2's skill and P3's panel.
   - **Solution search approach and hints:** the wall is rarely the compiler — it is a runtime/framework floor a hop crosses (Spring Boot 2→3 / jakarta, a too-old jacoco/ASM for new-JDK bytecode, Mockito/ByteBuddy). Take the cheap hops first; when a hop needs a framework major-bump, run the dedicated OpenRewrite migration recipe rather than hand-editing, then re-verify. If the framework migration cannot conserve tests, BAIL the request to P2's `feedback.jsonl` (a real corpus gap) rather than shipping a broken multi-hop. Re-enter per routed multi-step target.
   - **Reward:** merged multi-step PRs (primary); fraction of routed multi-step requests reaching the full-target PASS (secondary).
   - **Attention mechanism:** a P12 hand-off is the trigger; a hop's conservation failure pulls this problem to either fix-and-continue or BAIL; the terminal outcome writes to P12's scorecard and, on BAIL, to P2's `feedback.jsonl`.

## Principles cluster
**Cross-cutting — P15 and P16 govern HOW every problem above is solved; they are peers of the meta-problem P1 but numbered last so existing P2–P14 references are untouched. Imported from great-java-review (its P2/P3). Concrete instances already live in P3's "No budgets or limits (RLVR)" clause (a P15/P16 application) and P5's `header_up -Authorization` boundary-isolation (the P16 carve-out).**

15. **Problem (no pre-defense):** meet a problem when it occurs, not in advance.
   - **Value:** "speculative armor — a guard, null-check, retry, fallback, or abstraction for a failure that has NOT happened — is dead weight" that obscures where code actually breaks and multiplies untested paths. Code remains smallest and most honest when each defense addresses an observed failure. (Seneca: we suffer more in imagination than in reality.)
   - **Contract and constraints** *(operator-only)*: do NOT pre-defend. Write the direct path; let an unhandled case actually OCCUR — a crash, a real trace — before you guard it, then fix its ROOT (P16), not a blanket mask. No "just in case" null-checks/defaults/retries for inputs that cannot yet arise; no abstraction for a second caller that doesn't exist; no config knob nobody asked for. The bar to ADD a defense is a REPRODUCED failure, not a hypothesis. This is also why we do not pre-LIMIT: a cap, a short timeout, or an output truncation pre-arms against a hypothetical and in practice strangles the real path — "set timeouts effectively to infinity (a year), ALWAYS send the FULL data never a truncated tail, and never cap size/depth/reasoning." A limit is a guard you have not earned (kafka#17565's reasoning budget → 26k of noise; an 8000-char output cap once starved the model of the compile error it needed). Complement of P16: P15 is BEFORE the failure (don't pre-arm), P16 is AFTER (root-cause, don't mask). Same single carve-out as P16 — isolating a transient EXTERNAL fault at a boundary so it can't destroy unrelated work; hardening an OBSERVED failure class (keepalive/timeout after a real hang) is met-when-it-occurred, not pre-defense.
   - **Solution search approach and hints:** "if you can't name the trace a guard prevents, delete the guard." A diff that GROWS to handle the imagined is the smell; the direct path is usually smaller.
   - **Reward:** the direct path shipped; zero guards without a matching observed failure.
   - **Attention mechanism:** a "just in case" comment; a null-check on a value that can't be null yet; a retry/fallback for a path that has never failed; an abstraction with a single caller "for the future."
16. **Problem (bug fixing):** reproduce, root-cause, fix the cause, rerun clean — never a workaround.
   - **Value:** "a bug fixed at its root removes a whole failure class and leaves the code smaller; a guard/special-case hides the cause and accretes cruft." The discipline keeps the harness honest enough to trust its own measurements.
   - **Contract and constraints** *(operator-only)*: when something breaks — (1) REPRODUCE (a real trace); (2) find the ROOT mechanism, not the symptom; (3) fix the ROOT so the error cannot occur — NOT a `try/except`, "treat as X on failure", retry, or cap that lets the broken path keep running; (4) RERUN CLEAN; (5) clean up the consequences. A workaround is acceptable only as a labelled, tracked stopgap. **One exception: boundary fault-isolation** — catching a TRANSIENT EXTERNAL fault (a vLLM 400 on malformed tool-call JSON) at the agent boundary so it doesn't destroy unrelated work is tolerance, not masking. The test: does the `try/except` hide a root cause we could fix (forbidden) or isolate an external flake so good work survives (allowed)?
   - **Solution search approach and hints:** trace on the host before theorising. "When a fix ADDS code (a guard/branch/retry), suspect you are masking; the root-cause fix usually REMOVES code" (the redundant context, the wrong path).
   - **Reward:** less code and fewer workarounds after the fix than before; zero masks left in the resolved path; the rerun is clean.
   - **Attention mechanism:** the diff is the channel — a `try/except` that swallows, a "default to X on failure", a retry around a deterministic error, or a cap that hides an overflow, each signals the symptom was treated, not the cause.
